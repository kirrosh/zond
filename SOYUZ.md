# СОЮЗ — семейство инструментов

Два инструмента, разные задачи, один общий принцип устройства.

**zond** — сканер гигиены публичного API (проверяет чужой/свой работающий API против спеки).
**Spectr** — генератор человеческой документации для эндпоинтов .NET-сервиса (из исходников).

Их роднит не код, а **доктрина харнеса**: инструмент детерминирован и «туп», всё
суждение — снаружи, у агента. Инструмент выдаёт факты (**dump**), агент думает
(**reason**), агент возвращает структурированные изменения (**apply**). LLM либо не
внутри инструмента вообще (zond), либо заперт в узкой роли «отдай только JSON»
(Spectr). Оба раздают Claude-скиллы как интерфейс и держат чистое ядро отдельно от
тонкой CLI-обёртки. Это соглашение, а не общая библиотека — кода они сегодня не
делят (разные рантаймы).

## zond — сканер гигиены API

### Задача
Прогнать работающий API против его OpenAPI-спеки и найти расхождения контракта,
дыры доступа и проблемы состояния — так, чтобы этим могла пользоваться маленькая
команда без security-инженера. Позиционирование: **hygiene scanner, не bounty-тул**;
нет доказательства — нет high severity.

Что делает конкретно:
- **conformance** — ответы соответствуют схеме спеки (коды, тела, типы);
- **security/stateful** — проверки доступа и состояния (например, mass-assignment
  через активные пробы);
- **coverage** — какая доля операций спеки реально покрыта;
- **generate** — генерация тест-сьютов из OpenAPI;
- **report** — вывод в console/json/junit/ndjson/SARIF, HTML-отчёты, curl-репро.

### Как устроен
- **Bun + TypeScript**, собирается в один бинарь `dist/zond` (`bun build --compile`).
- CLI на **commander**; каждая команда — отдельный модуль `src/cli/commands/*.ts`
  (`run`, `check`, `coverage`, `generate`, `probe`, `audit`, `report`, `init`…).
- Ядро `src/core/`: `parser/` (YAML-сьюты), `runner/` (HTTP-исполнение поверх
  `fetch` с ретраями), `checks/` (каталог depth-проверок), `probe/`, `generator/`,
  `reporter/`+`exporter/`, `severity/`+`classifier/` (калибровка серьёзности),
  `coverage/`, `secrets/`, `workspace/`.
- **Валидация — zod (v4)** как единый источник истины; из zod же генерится JSON Schema.
  Состояние прогонов — **SQLite** (`.zond/zond.db`).
- Конфиг-модель per-API: `.env.yaml` (baseURL, timeouts, секреты), `.api-fixtures.yaml`
  (фикстуры), `zond.config.yml` (воркспейс). Иерархия дефолтов:
  CLI-флаг → `.env.yaml` → `zond.config.yml` → хардкод.
- Интерфейс для агента: скиллы `init/templates/skills/` (`zond.md`, `zond-triage.md`,
  `zond-checks.md`) — их раскладывает `zond init`.

### Что дополняет и что заменяет
| Инструмент | Отношение zond |
|---|---|
| **Schemathesis / Dredd** (spec-driven conformance) | **Заменяет** для малой команды: то же покрытие контракта плюс security/stateful и agent-driven триаж. |
| **42Crunch / StackHawk** (коммерческие API-security сканеры в CI) | **Заменяет** в OSS/small-team сегменте — тот же класс задач без ценника и без выделенного security. |
| **Postman / Newman** (ручные коллекции) | **Дополняет**: zond генерит сьюты из спеки автоматически, а не руками. |
| **Spectral / Stoplight** (статический lint OpenAPI) | **Дополняет**: Spectral проверяет саму спеку на диске, zond проверяет **живой** API против неё. |
| **Burp Suite / OWASP ZAP** (тяжёлый DAST/пентест) | **Не заменяет** — стоит ниже: лёгкий спек-first скан для команд без пентестера; глубокий ручной пентест остаётся за ними. |

## Spectr — генератор апидоков .NET

### Задача
Из исходников одного эндпоинта .NET-сервиса (Clean Architecture + CQRS/MediatR)
собрать **человеческую** документацию: markdown по шаблону ApiDocs, две диаграммы
PlantUML (Schema взаимодействия + Algorithm потока), примеры и раздел «Основной
сценарий». Плюс проверить готовую генерацию и опубликовать её. Цель — снять с
разработчика ручное писание апидоков и рисование диаграмм.

Что делает конкретно:
- **method/controller/service** — генерация дока для метода / контроллера / всего сервиса;
- **extract** — построение факт-шита из кода (единственный источник фактов);
- **lint** — детерминированные автопроверки полноты вывода;
- **critique** — LLM-критик полноты, привязанный к коду (консенсус-голосование);
- **sync/publish** — раскладка в репозиторий сервиса и публикация в Confluence.

### Как устроен
- **Node + tsx**, оркестрация — тонкий bash-диспетчер `bin/spektr` + `scripts/*.sh`
  (единый процесс не держится; шаги склеены как пайплайн).
- Конвейер `method`: **extract → refine → render**.
  1. `extract/roslyn.ts` зовёт **.NET Roslyn-экстрактор** (пребилт-бинарь или
     `dotnet run`) → JSON-факт-шит;
  2. `refine.ts` через **LLM-CLI `cn`** шлифует только подписи алгоритма
     (контракт JSON↔JSON, LLM не пишет puml);
  3. `diagram.ts` + `md.ts` **детерминированно** рендерят PlantUML и markdown из
     факт-шита; `plantuml.jar` (Java) превращает `.puml` → `.svg`.
- **LLM заперт в клетке**: нет SDK Anthropic/OpenAI, только внешний `cn` через
  единственную точку `src/llm.ts` (ретраи + rate-limit backoff). Топологию диаграмм
  и структуру md держит код, не модель. Промпты — в `src/resources/`.
- **Схемной валидации (zod/JSON Schema) нет** — типы в `src/types.ts`, а корректность
  вывода ловят детерминированные `lint.ts`/`algo-graph.ts`/`check-corpus.ts`.
- Конфиг — через `.env` (`DOCGEN_*`, `CONFLUENCE_*`); Альфа-специфика (имена сервисов,
  движок БД) вынесена из кода в данные (`default-profile.json`).
- **Feedback-loop**: находки на метод копятся в `feedback/<Method>.md` с маркер-блоками
  (`LINT:START…`, `CRITIQUE:START…` — перезаписываются инструментом; ручные секции
  «Баги» и «Правки скриптов» инструмент не трогает). Служит входом для правок самого
  генератора.

### Что дополняет и что заменяет
| Инструмент | Отношение Spectr |
|---|---|
| **Ручное писание апидоков в Confluence** | **Заменяет**: генерит текст и диаграммы из кода и публикует в Confluence сам. |
| **PlantUML / mermaid руками** | **Заменяет**: диаграммы Schema/Algorithm строятся детерминированно из факт-шита. |
| **DocFX / Doxygen** (reference из XML-комментариев) | **Заменяет** для нарратива эндпоинта: богаче — поток алгоритма и sequence-диаграммы, LLM-шлифовка формулировок. |
| **Swashbuckle / Swagger UI** (машинный OpenAPI-контракт) | **Дополняет**: Swagger описывает контракт для машин, Spectr — человеческий рассказ «как метод работает». Разные слои. |
| **Mintlify / Redocly** (порталы документации) | **Дополняет**: это слой представления; Spectr отдаёт им контент. |

## Одной таблицей

| | **zond** | **Spectr** |
|---|---|---|
| Домен | Гигиена работающего API | Документация .NET-эндпоинта |
| Вход | OpenAPI-спека + живой API | Исходники C# (Roslyn) |
| Выход | Отчёты, SARIF, покрытие, тесты | md + PlantUML svg + примеры |
| Рантайм | Bun, один бинарь | Node + tsx + bash |
| Валидация | zod / SQLite | детерминированные линты, типы |
| Роль LLM | нет внутри (агент снаружи) | заперт: только JSON-подписи |
| Общее | харнес dump→reason→apply, скиллы-интерфейс, чистое ядро ↔ тонкая CLI, config-as-data, feedback-loop | ← то же |

## Общая часть — статус
Сегодня общее у СОЮЗ — **доктрина и словарь**, не код (дублирования нет: у zond нет
LLM-обёртки by design, у Spectr нет zod/HTTP-слоя). Общий пакет извлекать по правилу
трёх — когда один и тот же код скопируется дважды или придёт третий аппарат семейства.
