# Auditing Vercel's OpenAPI spec with a coding agent, live

> **A coding agent ran zond against Vercel's official 332-endpoint API on a throwaway account and caught a live HTTP 500, ~150 operations whose real status codes the spec never declares, and ~45 create endpoints that reject the very bodies the spec calls valid — while sending zero requests to the account-deletion, billing, or domain-purchase endpoints that share the same spec.** Zero security findings. The story isn't a breach — it's how much an autogenerated spec drifts from the API it describes, and how carefully a live audit has to scope itself when the target has no sandbox.

## What we did

- Target: Vercel REST API, official spec ([`openapi.vercel.sh`](https://openapi.vercel.sh/), OpenAPI 3.0.3, ~9 MB, autogenerated).
- Tool: zond 0.27.1, driven by a coding agent following the `/zond-scan` convention, live mode.
- Mode: **live** on a throwaway Hobby-tier account. Vercel has no test-mode/sandbox API, so "live" is the real account — which forced a conservative mutation policy (below).
- Scope: 332 endpoints total. Read-and-boundary depth across a 173-operation safety-scoped subset; read-only smoke across the whole GET surface; mass-assignment + SSRF/CRLF/redirect probes on 21 create/update endpoints.

## What we found

**One real server error.** `GET /v3/events` returned HTTP 500 four times under the boundary-value fuzzing phase. Honest asterisk: a single follow-up manual call returned 200, so it's intermittent, not a hard 500 — but four real 5xx wire responses is evidence, and the spec declares no 500 for that operation at all. That's the one finding that's a server bug, not a doc bug.

**The rest is contract drift, and it's systemic:**

1. **~150 operations return status codes the spec never declares.** 52 by-id endpoints return an undeclared `404`; 146 return an undeclared `405` on unsupported methods. A client generated from this spec has no branch for the single most common error a by-id endpoint produces.

2. **~45 "create" endpoints reject bodies the spec calls valid.** `POST /v2/projects/{p}/checks` requires `source.kind` to equal a constant the spec types as a free-form string. `POST /v13/deployments` caps `serverlessFunctionRegion` at 4 characters with no `maxLength` in the schema. The spec is looser than the implementation — so codegen-happy-path bodies 400 on the first try.

3. **Silent tolerance of malformed pagination cursors** on `feature-flags/flags` — 2xx on garbage cursor input. Minor, but a consistency gap.

**Zero of the scary stuff:** no auth bypass with a data leak, no mass-assignment (every privilege-injection probe was blocked by the account being empty/unentitled before it could even test), no SSRF/CRLF/open-redirect. Honest reporting means saying that plainly.

## The honest asterisks (why this is a *hygiene* tool, not a bug-bounty flex)

- **honest-2xx was ~80% within the exercised scope (138/173 operations), ~45% across the full 332.** The gap between those two numbers is the whole point. It isn't "the scan failed on half the API" — it's that we *deliberately never sent* 153 mutation requests, because those endpoints include `DELETE /v1/user` (deletes the account), `POST /v1/billing/buy` (real money), `POST /v1/registrar/domains/{d}/buy` (buys a domain), and `POST /v1/security/attack-mode` (flips production security posture). On an API with a sandbox you'd fire all of them at a test tenant. Vercel has no sandbox, so the responsible move was a hand-audited allowlist of ~10 resource groups that self-create → act → self-delete, and a hard no-go list for everything that spends money or can't be undone.

- **The generated test suite itself would have deleted the account.** `zond generate` emitted a `smoke-user-unsafe.yaml` containing `DELETE /v1/user`, and `smoke-teams-unsafe.yaml` that PATCHes whatever real team id sits in the fixture file. A naive `zond run apis/vercel/tests` in live mode sends those. Catching that *before* running — not after — is the difference between a hygiene scan and an incident. (Filed as a zond safety-gap, not a Vercel bug — and fixed in zond the same day.)

- **The raw finding count was 1248 static + ~450 runtime — but the signal is a handful.** After calibration: 1 HIGH (the 500), three status/body-drift rollups, ~335 real-but-LOW spec violations, ~900 INFO doc-style gaps. We report the rollups, not the 1700. Inflated counts are noise.

- **Mass-assignment came back all-INCONCLUSIVE, and that's honest too.** Every injection probe needs a valid baseline create first; on an empty Hobby account, every create 400/403/404'd (payment required, not entitled, parent doesn't exist) before any `is_admin: true` could be tested. We report "couldn't test," not "clean."

## Why it matters for a small team

Vercel has a spec-tooling budget you don't. If *their* autogenerated spec omits 404s on 52 endpoints and 405s on 146, and models 45 request bodies looser than the server enforces, a hand-maintained spec on your 30-endpoint internal API drifts more. And the second lesson is sharper: when you point a live scanner at a real account, the scanner's own generated tests are part of your blast radius. The value here is an agent that (a) caught the drift and the flaky 500 in one pass, and (b) recognized the account-deletion endpoint in its own test suite and scoped around it — before running.
