# Быстрый старт: покрытие API тестами через AI-агента

AI-агент (Claude Code) + zond = автоматическое покрытие вашего API тестами по OpenAPI-спецификации. Вы даёте задание на естественном языке, агент сам генерирует тесты, запускает их и показывает результаты.

---

## Что понадобится

- **Node.js 18+** — [nodejs.org](https://nodejs.org/) (проверить: `node --version`)
- **Claude Code** — CLI-инструмент от Anthropic (установим ниже)
- **OpenAPI-спецификация** вашего API — файл `openapi.json` или `openapi.yaml`, либо URL

---

## Шаг 1. Создать папку проекта

Создайте отдельную папку — в ней будут храниться тесты, конфигурация окружения и база данных с результатами запусков.

```bash
mkdir my-api-tests
cd my-api-tests
```

Скопируйте сюда OpenAPI-спецификацию вашего API (или запомните URL, где она доступна):

```bash
cp ~/path/to/openapi.json .
```

---

## Шаг 2. Установить и запустить Claude Code

> Если Claude Code уже установлен — пропустите установку и переходите к запуску.

Установите Claude Code глобально (один раз). Подробная инструкция по установке и настройке: [code.claude.com/docs/en/setup](https://code.claude.com/docs/en/setup)

```bash
npm install -g @anthropic-ai/claude-code
```

Запустите с флагом автоматического подтверждения всех действий:

```bash
claude --dangerously-skip-permissions
```

> **Что делает этот флаг?** Обычно Claude Code спрашивает разрешение на каждое действие — создание файлов, запуск команд и т.д. С этим флагом агент работает автономно, без ручных подтверждений. Используйте его только в изолированных папках, где нет важных данных.

---

## Шаг 3. Подключить zond

Есть два способа — выберите один.

### Способ А: Claude Code плагин (рекомендуется)

Плагин даёт MCP-инструменты + скиллы + slash-команды (`/zond:api-test`, `/zond:api-coverage`):

```
/plugin marketplace add kirrosh/zond
```

Затем установите плагин:

```
/plugin install zond@zond-marketplace
```

### Способ Б: MCP-сервер (для Cursor, Windsurf и других редакторов)

Если вы используете не Claude Code, а другой редактор с поддержкой MCP:

```json
{
  "mcpServers": {
    "zond": {
      "command": "npx",
      "args": ["-y", "@kirrosh/zond@latest", "mcp"]
    }
  }
}
```

Добавьте эту конфигурацию в настройки MCP вашего редактора. Подробнее: [docs/mcp-guide.md](mcp-guide.md).

---

Проверьте, что подключение работает — в приветственном сообщении должен появиться `zond` в списке MCP-серверов, или выполните `/mcp` для диагностики.

---

## Шаг 4. Дать задание агенту

Теперь дайте агенту первое задание. Если установлен плагин (Способ А), можно использовать slash-команду:

```
/zond:api-test openapi.json
```

Или дайте задание на естественном языке — сгенерировать безопасные smoke-тесты (только GET-запросы, которые не меняют данные):

```
Покрой API из файла openapi.json безопасными smoke-тестами.
Начни только с GET-запросов (safe: true).
Запусти тесты и покажи результаты.
```

> **Подсказка:** Не обязательно писать такой подробный промпт. Например, `Покрой openapi.json smoke-тестами` тоже сработает.

> **Что произойдёт?** Агент выполнит несколько шагов через CLI:
>
> 1. `zond init --spec openapi.json` — зарегистрирует ваш API, создаст структуру папок и файл `.env.yaml`
> 2. `zond generate openapi.json --output apis/myapi/tests/` — сгенерирует YAML-тесты
> 3. `zond run apis/myapi/tests/ --safe` — запустит только GET-запросы
> 4. Покажет сводку: сколько тестов прошло, сколько упало и почему

---

## Шаг 5. Настроить авторизацию

Если ваш API требует авторизации, часть тестов упадёт с ошибкой `401 Unauthorized`. Агент увидит это и спросит токен. Есть два варианта:

### Вариант А: Сказать агенту прямо в чате

```
Токен для авторизации: Bearer sk-your-token-here
Сохрани его в .env.yaml и перезапусти упавшие тесты.
```

### Вариант Б: Отредактировать файл вручную

После шага 4 агент создал файл `.env.yaml` в папке вашего API (например, `apis/myapi/.env.yaml`). Откройте его и добавьте токен:

```yaml
base_url: https://api.example.com
auth_token: Bearer sk-your-token-here
```

Затем попросите агента перезапустить тесты:

```
Перезапусти тесты с обновлённым окружением.
```

> **Важно:** Файл `.env.yaml` автоматически добавлен в `.gitignore` — ваши токены не попадут в git.

---

## Шаг 6. Посмотреть результаты в Web UI

Попросите агента запустить веб-дашборд:

```
Запусти веб-интерфейс zond чтобы я мог посмотреть результаты.
```

> **Что произойдёт?** Агент вызовет `manage_server(action: 'start')` и покажет ссылку — обычно `http://localhost:8080`.

Откройте ссылку в браузере. Вы увидите:

- **Health strip** (верхняя панель) — круговая диаграмма покрытия, количество пройденных/упавших/пропущенных тестов, прогресс-бар
- **Вкладка Endpoints** — все эндпоинты из OpenAPI-спецификации с цветными индикаторами статуса:
  - зелёный — тесты проходят
  - красный — тесты падают
  - серый — нет тестов или не запускались
  - предупреждения: `DEPRECATED`, `NO SCHEMA`, `MISSING EXAMPLES`
- **Вкладка Suites** — список YAML-файлов с тестами; кликните на suite чтобы увидеть каждый шаг, его assertions, captures и response body для упавших
- **Вкладка Runs** — история запусков с прогресс-барами; кликните на run для детальной информации, экспорт в JUnit/JSON

## Что дальше

После smoke-тестов вы можете попросить агента делать более сложные вещи. Вот примеры промптов:

### Проверить покрытие

Если установлен плагин:

```
/zond:api-coverage openapi.json
```

### Починить сломанные тесты

```
Посмотри на упавшие тесты в последнем запуске.
Разберись в чём проблема и предложи как исправить.
```

> Агент вызовет `diagnose_failure`, увидит ответы сервера (status code, response body) и объяснит причину: неверная схема, отсутствующий ресурс, проблема с авторизацией. Затем предложит конкретные правки в YAML-тестах.

### Покрыть CRUD-операции (с откатом данных)

```
Сгенерируй CRUD-тесты для всех эндпоинтов.
Каждый тест должен создавать данные в начале и удалять их в конце,
чтобы API оставался в исходном состоянии.
Сначала покажи запросы через dry-run, не отправляя ничего.
```

> Агент сгенерирует тесты с паттерном create → read → update → delete. Сначала покажет все запросы через `dryRun: true` — вы увидите URL, метод и тело каждого запроса без реальной отправки. Только после вашего подтверждения агент запустит тесты.

### Увидеть потенциально проблемные эндпоинты

```
Покажи анализ покрытия: какие эндпоинты не покрыты тестами,
какие помечены как deprecated, у каких нет схемы ответа.
```

> Агент вызовет `coverage_analysis` и покажет полную картину: процент покрытия, эндпоинты без тестов, предупреждения из спецификации. Это поможет расставить приоритеты — какие тесты писать следующими.

### Сравнить два запуска (регрессия)

```
Сравни последний запуск тестов с предыдущим.
Что сломалось, что починилось?
```

> Агент вызовет `compare_runs` и покажет: новые падения (регрессия), исправленные тесты, изменения во времени ответа.

### Добавить тесты в CI/CD

```
Сгенерируй GitHub Actions конфиг для автоматического запуска smoke-тестов.
```

> Агент вызовет `ci_init` и создаст `.github/workflows/zond.yml` с расписанием, ручным триггером и отчётами.

## Итого: что получится

```
my-api-tests/
├── zond.db                ← база данных с результатами запусков
└── apis/
    └── myapi/
        ├── openapi.json   ← ваша OpenAPI-спецификация
        ├── .env.yaml      ← переменные окружения (base_url, токены)
        ├── .gitignore     ← .env*.yaml исключены из git
        └── tests/
            ├── smoke.yaml ← GET-тесты
            └── crud.yaml  ← CRUD-тесты
```

Все тесты — обычные YAML-файлы, которые можно редактировать вручную, хранить в git и запускать в CI.
