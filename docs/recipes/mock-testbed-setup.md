# Recipe: mock-API testbed for m-20 stateful probes

> m-21 ARV-193. Self-contained, intentionally buggy API used as the
> regression-floor for the four m-20 stateful probes
> (`cross_call_references`, `idempotency_replay`, `pagination_invariants`,
> `lifecycle_transitions`). If a probe stops detecting the class of bug
> it owns ‚Äî anti-FP regression, severity downgrade, evidence-shape
> drift ‚Äî this recipe + `tests/regression/mock-testbed.test.ts` is the
> canary.

## What this recipe answers

- Are all four m-20 stateful probes still firing on a known-buggy
  API after a refactor?
- For a probe under development: does it produce a HIGH finding with
  the expected `evidence.kind` on the canonical bug for its class?
- For a contributor onboarding a fifth stateful probe: where do I
  plant a bug + an assertion so my probe joins the regression floor?

## What's inside

```
apis/_mock/
‚îú‚îÄ‚îÄ spec.json                    # OpenAPI 3.0.3 ‚Äî single resource: widgets
‚îú‚îÄ‚îÄ .api-resources.yaml          # CRUD layout (regenerated by `zond add-api`)
‚îú‚îÄ‚îÄ .api-resources.local.yaml    # idempotency / pagination / lifecycle / seed_body
‚îî‚îÄ‚îÄ server.ts                    # Bun.serve() ‚Äî 4 intentional bugs, no external deps
```

Why two yaml files: `add-api` rebuilds `.api-resources.yaml` from the
spec on every run, so user-curated overrides have to live in the
sibling `.local.yaml` (under top-level `patches:`) where `refresh-api`
won't touch them. Both files in `apis/_mock/` are checked in so the
directory is self-documenting.

`server.ts` exports `startMockServer({ port })` so the regression test
can boot it on an ephemeral port; `bun apis/_mock/server.ts` runs it
standalone on `http://127.0.0.1:7717` for manual exploration.

## The four declared bugs

Every bug below maps 1:1 to one probe. The mapping is the contract
this testbed defends.

| Bug                                     | Probe                       | Expected `evidence.kind` |
| --------------------------------------- | --------------------------- | ------------------------ |
| B1 POST echoes `color`, GET drops it    | `cross_call_references`     | `state_not_persisted` (field `color`) |
| B2 cursor is off-by-one (page B[0] == cursor) | `pagination_invariants` | `duplicate_items` |
| B3 `Idempotency-Key` is ignored         | `idempotency_replay`        | `duplicate_resource` |
| B4 `POST /widgets/{id}/publish` no-ops  | `lifecycle_transitions`     | `wrong_expected_state` |

When you change the server, change the matching expectation in
`tests/regression/mock-testbed.test.ts` in the same commit. Never
silence one half ‚Äî the whole point is that they move together.

## Run the testbed against `zond`

The testbed is registered in the workspace like any other API. The
manifest already lives in `apis/_mock/`, so `add-api` just stamps the
DB row.

```bash
# 1. boot the buggy server (separate terminal, or background it)
bun apis/_mock/server.ts            # listens on http://127.0.0.1:7717

# 2. register it in the workspace
zond add-api _mock \
  --spec apis/_mock/spec.json \
  --base-url http://127.0.0.1:7717

# 3. run only the stateful probes against it
zond checks run --api _mock --check stateful
```

Note the canonical alias: `--check stateful` selects the four m-20
checks. `--phase stateful` is **not** a valid phase (`--phase` accepts
`examples`, `coverage`, `all` only) ‚Äî `checks run` rejects it with a
hint pointing here. You can also list the four ids explicitly:

```bash
zond checks run --api _mock \
  --check cross_call_references,idempotency_replay,pagination_invariants,lifecycle_transitions
```

Expected output: **4 HIGH findings**, one per probe, on the `widgets`
resource. Exit code 1 (HIGH/CRITICAL findings).

## Run the regression test

The CI test is hands-off ‚Äî it boots the server on an ephemeral port,
calls `runChecks` programmatically with the resource configs from the
yaml manifest, and asserts the 4-of-4 mapping table above.

```bash
bun test tests/regression/mock-testbed.test.ts
```

It runs as part of `bun test` in CI (`.github/workflows/ci.yml`); no
extra wiring needed.

## Adding a fifth probe + bug

When a future milestone lands a fifth stateful probe (e.g. `etag_concurrency`):

1. Plant a minimal bug in `apis/_mock/server.ts`, behind a comment that
   names the probe and the expected `evidence.kind`.
2. Add the matching declaration to `apis/_mock/.api-resources.yaml` if
   the probe needs yaml opt-in (lifecycle-style); otherwise rely on
   spec autodetect (idempotency / pagination style).
3. Add the probe id to `STATEFUL_CHECKS` in
   `tests/regression/mock-testbed.test.ts` and add the per-evidence
   assertion block.
4. Update the table in this recipe.

Keep the bug minimal ‚Äî one class of failure per probe. The point of
this testbed is to assert "the probe still fires on its canonical
target", not to model a realistic broken API.
