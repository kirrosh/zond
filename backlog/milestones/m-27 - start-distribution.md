## Цель

Довести zond до состояния **«чужой инженер поставил и за 5 минут получил первый
зелёный прогон»** по всем заявленным каналам, на чистой машине, без знания
внутренностей. Это не набор фич — это условие, чтобы выбранная ниша (hygiene
scanner для маленьких команд, decision-8) вообще увидела инструмент.

Продукт технически зрел после m-24/m-25; открытая граница — **упаковка и
онбординг**, а не движок. Новый функционал (schemathesis fuzz, новые
check-классы ARV-370/371) в m-27 **не** тащим — они форвардные, но играют на
переполненном поле; распределяются отдельной вехой.

decision-8 (positioning) и litmus test остаются в силе.

## Что уже стоит (базовая линия на 2026-07-09)

- Репозиторий публичный (`kirrosh/zond`).
- GH-релизы с бинарями `darwin-arm64`, `linux-x64`, `win-x64`.
- `install.sh` / `install.ps1` — curl-инсталлеры, тянут бинарь из latest-релиза.
- README v2 (ARV-364, Done) — позиционирование + quickstart.
- `bun build --compile` → self-contained `dist/zond`.

## Дыры, которые закрывает m-27

1. **npm-канал по факту сломан.** `bin.zond → src/cli/index.ts` — сырой TS,
   требует установленный bun. `npm i -g @kirrosh/zond` у node-only юзера не
   запустится. Заявлено в DoD m-25 («npm i -g работает на чистой машине»), не
   выполнено.
2. **Матрица платформ дырявая.** Нет `darwin-x64` (Intel-маки) и `linux-arm64`.
   `install.sh` мапит их на несуществующие артефакты → 404 у части юзеров.
3. **brew-формулы нет вообще** (заявлена в ARV-365).
4. **Cold-start `init`** делает «ты — это я»-допущения; для чужого репо путь до
   первого прогона не выверен.
5. **Enterprise-трение** (ARV-367): corp CA spec-fetch + apiKey auth hint.
6. **Нет проверки «на чистой машине»** — что все каналы реально работают у
   стороннего инженера, а не только на машине автора.

## Bucket A — Каналы установки работают на чистой машине

- **npm-канал (node-only).** Развязать `bin` от bun: bin становится тонким
  лаунчером, `postinstall` тянет платформенный бинарь из релиза (модель
  esbuild/turbo) ИЛИ переносим bin на скомпилированный бинарь. Критерий:
  `npm i -g @kirrosh/zond && zond --version` на машине без bun. *(re-scope ARV-365)*
- **brew tap + формула.** `brew install kirrosh/tap/zond` тянет release-бинарь
  с checksum. Tap-репо или Formula в основном репо.
- **Матрица платформ.** Добавить `darwin-x64` + `linux-arm64` в релиз;
  `install.sh`/`.ps1` — либо покрывают все 5 таргетов, либо деградируют с
  внятным сообщением вместо 404.
- **Release-пайплайн воспроизводим.** Один прогон кросс-компилит все таргеты,
  подписывает (darwin codesign уже есть), считает checksums, аттачит к
  `gh release`, публикует npm, бампает brew. Документированный скрипт/CI, не
  ручная сборка per-arch.

## Bucket B — Cold-start онбординг (чужой репо → первый зелёный прогон)

- **`zond init` cold-start UX.** Убрать «ты — это я»-допущения; выверенный путь
  install → init → doctor → первый `audit --safe` за <5 минут на чужом репо.
  Первый экран говорит, что заполнить в `.env.yaml` и куда идти дальше.
  *(вынести из хвоста ARV-365 в свою задачу)*
- **Enterprise/on-prem трение** — ARV-367: corp CA (`--ca` / `NODE_EXTRA_CA_CERTS`)
  для spec-fetch за self-signed, apiKey-auth hint при docgen-скане. *(re-home)*

## Bucket C — Trust & discoverability

- **README/landing distribution-pass.** Матрица установки (curl/npm/brew/win),
  60-секундное демо (asciinema/gif), явная нота про safe-by-default (что zond
  НЕ сломает прод на первом прогоне), one-liner позиционирования для незнакомца.
- *(опционально, deferred)* GitHub Action-обёртка как CI-канал дистрибуции —
  решить в m-27, реализовать при спросе.
- Launch/маркетинг (Show HN, awesome-* листы) — **вне ядра m-27**, отдельный
  трек; здесь только готовность артефактов к такому запуску.

## Bucket D — Distribution validation (DoD-гейт)

- **Clean-machine E2E.** Свежий контейнер/VM на каждый канал (curl, npm, brew,
  win): install → `zond init` в чужом публичном репо → первый `audit` с нулевым
  знанием внутренностей. Зафиксировать каждую точку трения; трение = баг канала.

## Definition of done

- A: `npm i -g` **и** `brew install` **и** `curl | sh` дают рабочий `zond` на
  чистой машине для всех 5 таргетов; релиз собирается одним воспроизводимым
  прогоном.
- B: незнакомый инженер доходит от install до первого зелёного `audit --safe`
  на своём репо без ручной правки внутренностей и без чтения исходников;
  enterprise-CA путь работает.
- C: README даёт незнакомцу понять «что это, зачем, как поставить, безопасно ли»
  за один экран; demo-артефакт есть.
- D: E2E-прогон на чистой машине пройден по каждому каналу, лог трения закрыт.
