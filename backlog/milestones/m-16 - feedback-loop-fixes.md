## Description

Зонтичный майлстоун для багов и UX-папереркатов, которые всплывают при
прогоне zond против реальных публичных API через парный ralph-loop
(`/zond-fb-tester` ↔ `/zond-fb-fixer`). Tester живёт в
`~/Projects/zond-test`, гоняет API, складывает структурированный feedback
в `~/Projects/zond-test/.fb-loop/rounds/`. Fixer (этот воркфлоу) читает
feedback, заводит сюда TASK-ы и чинит.

В отличие от m-1…m-15 — это **не плановый** скоуп с критическим путём,
а постоянно пополняемый bucket из эмпирических наблюдений. Майлстоун
закрывается не «когда всё done», а когда раунды feedback'а перестают
давать новых high/medium находок.

## Источник задач

Каждая задача в этом майлстоуне создаётся фиксером из конкретного
finding'а в `feedback-NN.md`:

```
Source: feedback round NN, finding F<k>, class <class>
Repro: <команда>
Expected: ...
Actual: ...
Log: ~/Projects/zond-test/.fb-loop/rounds/raw-NN.log
```

Метки: `feedback-loop`, `api-<slug>` (например `api-resend`,
`api-sentry`).

## Приоритизация

Берётся напрямую из severity finding'а:

- `definitely_bug` high-severity → priority **high** (фиксится в том же
  раунде).
- `likely_bug` / `missing-feature` без обхода → **medium** (добирается
  если high'ов мало).
- `quirk` / `ux-papercut` / косметика → **low** (копится, разгребается
  пачкой).

## Принципы

- **Маленькие коммиты, узкие диффы.** Один TASK = один коммит
  `TASK-NNN: <subject>`. Не смешивать несколько finding'ов в одну
  правку — verify-режим тестера должен матчить feedback ↔ fix
  один-к-одному.
- **Anti-FP first.** Если finding похож на ложное срабатывание самого
  zond'а (а не на баг в API под тестом) — добавить fixture-тест на
  регрессию до фикса.
- **Не трогать API-баги.** Если finding на самом деле описывает кривость
  тестируемого API, а не zond — задача не заводится, finding уходит в
  `Skipped` секцию fix-report'а.
- **Бинарь после раунда.** После всех фиксов раунда — один
  `bun run build && cp ./dist/zond ~/.local/bin/zond`, чтобы тестер
  увидел свежий mtime и перешёл в verify.

## Не покрывает

- Плановые фичи из стратегий vector-2…5 — это m-17+.
- Глубокие архитектурные рефакторы — выносятся в отдельные
  тематические майлстоуны, даже если первым их обнаружил feedback-loop.
- Баги тестируемых API — не наша ответственность.

## Done-критерий

Майлстоун считается «зрелым» (не закрытым), когда:

1. Прогон `/zond-fb-tester` против незнакомого публичного API не
   приносит ни одного `severity: high` finding'а в первом раунде.
2. Доля `verified-from-prev` в новых feedback'ах стабильно > 80%
   (фиксы держатся, регрессий нет).
3. Все `low`-папереркаты раз в N раундов разгребаются пачкой, не
   копятся бесконечно.

Формального закрытия не предполагается — майлстоун живёт пока живёт
feedback-loop.
