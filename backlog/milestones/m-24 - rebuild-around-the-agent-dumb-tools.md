## Цель

Пересобрать zond как **набор детерминированных dumb-инструментов**, которыми
агент собирает test-suite так, как это делал бы человек — проходясь по
инструментам, а когда чего-то не хватает (фикстура, auth, значение) —
**дозапрашивая у пользователя**, а не угадывая эвристикой.

Всё хранится в **agent-friendly yaml**. По прогону zond умеет отдать
**сводку для разработчиков** и **сравнить два прогона** (diff — особенно
после изменения API).

## Что zond делает (детерминированно) vs агент (суждение)

| zond — механика | агент — суждение |
|---|---|
| послать запрос, записать ответ | что тестировать и в каком порядке |
| validate тело против схемы | что значит фикстура, чем её заполнить |
| сохранить прогон; **diff двух прогонов** | реальный ли баг, severity |
| сказать «фикстура X пустая / auth протух» | что дозапросить у пользователя |
| сгенерить сводку из сохранённых прогонов | вердикт для разработчиков |

## Убрать (источник баг-стрима — см. decision-9)

- авто-`discovery` фикстур (клал numeric-id в login-слот → ARV-334)
- `prepare-fixtures --seed --cascade` авто-создание ресурсов (1% на Stripe)
- `annotate auto` эвристики
- severity-калибраторы (severity — суждение агента)

## Оставить (чистая механика, дёшево, надёжно)

- отправка запросов + запись прогонов
- schema-validation
- checks/probes-движок как **инструмент** (агент решает, что запускать)
- coverage/diff как сравнение прогонов

## Definition of done

- Эвристический слой удалён; фикстуры/auth — агент+пользователь (дозапрос).
- Прогоны хранятся в yaml, пригодном агенту для сборки/сравнения.
- Есть команда сводки для разработчиков и команда diff двух прогонов.
- README/skills переписаны под новую форму (агент-оркестратор).
